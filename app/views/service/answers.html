Answers - accessibility issues for the register for a service

Start page

  1. Two H1 tags

    It is good practice to have only one H1 tag on a page. The H1 is the most important heading and tells you what the purpose of the overall page is. Screen reader users can use headers to navigate a page by skipping to them so having a good heading structure helps.

    Understanding WCAG regarding headers

    • 1.3.1 Info and Relationships
    • 2.4.1 Bypass Blocks
    • 2.4.6 Headings and Labels
    • 2.4.10 Section Headings

  2. 'NHS number' link not underlined

    Users may not perceive colour differences so would not identify this as a link. Links should be underlined or have some other non-colour visual distinction.

    Understanding WCAG regarding use of colour

    • 1.4.1 Use of Color

  3. 'Mental health services' link opens in new tab

    Opening a new tab (or window) when a link is activated can be disorienting for some users if they are not warned in advance. Users may be unaware that it has happened and won't be able to use the back button as there is no history in the new tab.

    Techniques for opening new tabs or windows

    • Opening new windows and tabs from a link only when necessary
    • Giving users advanced warning when opening a new window

What is your name?

  • First name label not associated with the input field

    The label tag is missing a 'for' attribute and the input field is missing the associated 'id' attribute. This means the function or purpose of the input field may not be presented to screen reader users. Form labels also provide visible descriptions and provides a larger clickable area for the input field, since clicking on the label will activate it and bring it to focus. This can be helpful for users with impaired motor control.

    Understanding WCAG regarding form labels and inputs

    • Using label elements to associate text labels with form controls

  • Visually hidden item you can tab onto

    There is a visually hidden input field that can be tabbed to. This might be required for example to pass data between pages. Although the input field is hidden off screen, it still receives focus. This could confuse keyboard users who are tabbing through the page because the focus would disappear.

    Understanding WCAG regarding focus

    • 3.2.1: On Focus

What is your date of birth?

  • Wrong page title

    Some screen readers (like JAWS) read the page title when a page loads. This page title is 'What is your name? - Register for our new service' which is incorrect as it should relate to what the page subject is, which in this case is asking what your date of birth is. The page title is the same as the previous page and has not been changed.

    Understanding WCAG regarding page titles

    • 2.4.2: Page Titled

  • No legend element

    We can visually tell that the input fields relate to the main heading by looking at the page. But, the question isn't linked up to the element in a way that makes sense for users who can't see. For radio buttons and checkboxes, it's important to have a legend element.

  • The inputs are marked up in a table element

    Use the table element for tabular data and not for layout puurposes.

  • Continue button not keyboard focusable

    You can still get past this page by pressing enter when on the radio buttons, but a user might not know this. They might (understandably) expect that the only way to progress is by submitting enter, and so feel trapped and abandon the service at this point. All interactive elements must be usabled with the keyboard. The fix would be removing the tabindex="-1"
